# 営業・CS支援アプリ 要件定義書

- 対象システム: アイドマ SU2.0 営業・CS支援アプリ(社内スタッフ向け + クライアントポータル)
- 版: 現行実装(as-built)に基づく初版
- 技術基盤: Next.js 14(App Router) + TypeScript + Supabase(Postgres / Auth / Storage) + Anthropic Claude API + Resend

---

## 1. システム概要

### 1.1 目的
アイドマが提供する営業支援サービス「SU2.0」の運用を効率化するため、以下を1つのアプリに統合する。

- クライアント企業ごとの案件(ヨミ表)・リード・次回アクションの一元管理
- 商談文字起こしからのAIによる議事録・フィードバック・スコアリング自動生成
- クライアントへの週次/月次定例報告の作成・送信
- クライアント自身による案件データの直接入力(社内の二重入力を廃止)

### 1.2 背景・業務フロー
SU2.0の支援フローは「初期設計→資料収集→案件管理表構築→営業資料整備→GPT設定→運用準備→運用」の7フェーズで進行し、運用フェーズでは毎週金曜にクライアントへヨミ件数・ヨミ金額を報告し、月次で定例MTGを実施する。本システムはこのフロー全体を`companies.support_phase`で可視化し、週次/月次報告を半自動化する。

### 1.3 利用者(アクター)

| アクター | 認証方式 | 主な用途 |
|---|---|---|
| 管理者(admin) | Google OAuth | 全機能・全クライアントの管理、ユーザー/プロンプト管理 |
| マネージャー(manager) | Google OAuth | adminとほぼ同等(自己昇格は不可) |
| 営業/CS/サポート担当(sales/cs/support) | Google OAuth | 自分がownerの企業・案件の編集、全企業の閲覧 |
| クライアント企業担当者 | 企業共有パスワード(独自セッション) | 自社の案件・リードの閲覧・入力、分析確認 |

## 2. 機能要件

### FR-1. 認証・アクセス制御

| ID | 要件 |
|---|---|
| FR-1-1 | スタッフはGoogle OAuthでログインする。事前に管理者がメールアドレスとロールを登録しておき、初回ログイン時に自動で紐付ける。 |
| FR-1-2 | ユーザーの有効/無効状態およびロールは、リクエストごとにDBを参照し即時反映する(JWTにキャッシュしない)。無効化されたユーザーは即座にアクセス不可となる。 |
| FR-1-3 | `/admin`配下はrole=admin/managerのみアクセス可能とする。 |
| FR-1-4 | クライアント企業には、企業ごとに共有のログインURL(slug)とパスワードを発行する。個人アカウントは発行しない。 |
| FR-1-5 | クライアントセッションはSupabase Authと独立した署名付きCookie(HMAC-SHA256、30日有効)で管理する。 |
| FR-1-6 | クライアントポータルのパスワードは、クライアント自身によるセルフリセットを提供しない(本人確認手段がないため)。リセット申請は担当スタッフへの通知のみとし、企業の存在有無に関わらず常に同じ成功応答を返す。 |
| FR-1-7 | 企業・案件・リード等の編集権限は「担当者(owner_user_id)本人」または「admin/manager」または「該当クライアント企業本人」に限定する。上記以外の担当者は編集できない。 |
| FR-1-8 | 自分自身のロール・有効状態は自分では変更できない(自己昇格防止)。 |
| FR-1-9 | 企業・案件データの閲覧は、ロールに関わらず全スタッフに開放する(担当外クライアントの情報も閲覧可能。編集のみ制限する)。 |
| FR-1-10 | クライアントポータルのログインは、同一slugで直近15分以内に8回失敗した場合、一時的にログインを拒否する(共有パスワード1つで運用するためのブルートフォース対策)。 |
| FR-1-11 | クライアントポータルのログインエラー(パスワード誤り・ロックアウト)は、ページ全体のエラー画面ではなくログインフォーム内にインライン表示する。 |

## 3. 非機能要件

### NFR-1. セキュリティ
- パスワードはハッシュ化して保存し、平文を保持しない。
- セッションは署名付きCookieで検証し、署名検証は定数時間比較で行う。
- 権限チェックはサーバーサイド(Server Action / APIルート)で必ず行い、クライアント側の表示制御に依存しない。
- CSV数式インジェクション(CWE-1236)対策として、危険な先頭文字を持つセルを無害化する。
- アップロードファイルは拡張子ホワイトリストとContent-Type固定により、意図しないコード実行を防止する。
- ロール・有効状態は毎リクエストDB参照とし、無効化・降格を即時反映する。
- クライアントポータルは共有パスワード運用のため、同一slugへの短時間の大量ログイン失敗を検知し一時ロックアウトする(FR-1-10)。
- Server Actionが本番ビルドでthrowしたエラーは、Next.jsの仕様によりメッセージが汎用文言に丸められクライアントへ届かない。ユーザーが実際に踏みうる検証エラー(ファイル形式不正、ログイン失敗等)は、throwではなく戻り値`{error: string}`として返す実装方針とする。

### NFR-2. 可用性・保守性
- メール送信基盤(Resend)・AI基盤(Anthropic)が未設定の環境でも、モックモードでアプリ全体が動作すること。
- 重要な操作(作成・更新・削除・生成・送信)は監査ログに記録すること。
- システム内部エラー(AI生成・メール送信・バッチ処理の失敗)はエラーログに記録し、管理画面から確認できること。
- システム内部エラーの発生は、管理画面の受動的な確認だけでなく、能動的なメールアラートでも検知できること(FR-9-6)。

### NFR-3. テスト
- 主要な業務フロー(認証、権限スコープ、案件⇄リード自動変換、次回アクションCRUD、レポート検索、セキュリティ対策)はE2Eテストで自動検証できること。
- E2Eテストは本番相当のデータベースに対して実行されることを前提とし、テスト自身が作成したデータのみを対象に後始末を行い、既存データに影響を与えないこと。
- E2Eテストは本番ビルド(`next build && next start`)に対して実行すること。開発モード(`next dev`)はコンパイル遅延によるタイミング起因の偽陽性・偽陰性が生じやすく、信頼できる検証結果にならない。
- E2EテストはChromium/Firefox/WebKitの3ブラウザで実行する。ただしWebKit(Windows版Playwrightビルド)はナビゲーション・ダウンロードイベント等の自動化が本質的に不安定な傾向があり(実行のたびに成功・失敗が入れ替わる既知の傾向。詳細はbacklog.md参照)、`retries: 1`を許容し、ネイティブHTML5 Drag and Dropのように自動化が原理的に対応できない操作は対象外とする。
- 実施者と検証者が同一である(独立したQAレビュー工程がない)ことは既知の限界として認識し、リスクの高い変更ほど複数の検証方法(自動テスト・手動ブラウザ操作・データ整合性確認)を組み合わせること。

### NFR-4. データ整合性
- ポリモーフィックな関連(生成レポートの対象種別、監査ログの対象種別)は、種別ごとの参照整合性をアプリケーション層で保証すること。
- 失注→リード自動変換は、同一案件について重複登録されないこと。

## 4. データモデル概要

| エンティティ | 主なフィールド | 備考 |
|---|---|---|
| `users` | name, email, role, status, auth_user_id | ロールはtext+CHECK制約(enum不使用) |
| `companies` | name, industry, contract_start/end, support_status, support_phase, owner_user_id, client_login_slug, client_password_hash, client_portal_enabled, notification_email, default_deal_category, default_lead_source | |
| `company_supporters` | company_id, user_id | 複数担当者アサイン用の中間テーブル |

## 6. 制約事項・前提条件

- クライアントポータルは企業単位の共有パスワードであり、企業内の担当者個人を識別しない。
- Row Level Security(RLS)は「認証済みユーザーは全件アクセス可」という最小限の設定であり、ロール別・担当者別の実質的なアクセス制御はすべてアプリケーション層(Server Action)で行う。
- 商談動画(一次/二次)はファイルアップロード対象外とし、Google Drive等外部サービスのURLを貼り付ける運用を前提とする。
- AIによるスコア・BANT自動算出値は「ドラフト」扱いであり、マネージャー確認済みかどうかを区別する状態を持つが、確認操作そのもののUIは本書執筆時点で別途確認が必要(現状把握の対象外)。
